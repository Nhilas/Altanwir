# Lint Configuration

This folder holds the linter configurations for this repository. Two linters are active:

| Tool | Config file | Scope |
|------|-------------|-------|
| [ruff](https://docs.astral.sh/ruff/) | `ruff.toml` | All Python notebook exports (`**/notebook-content.py`) |
| [sqlfluff](https://docs.sqlfluff.com/) | `.sqlfluff` | `Fabric/IGDBAnalytics.SQLEndpoint/` and `Fabric/IGDBAudit.Warehouse/` |

Both run automatically on push and pull request via `.github/workflows/lint.yml`.

---

## What is linted

### Python (ruff)
All `notebook-content.py` files under `Fabric/`. These are Fabric notebook exports — Fabric serialises notebooks to Python with `# CELL **`, `# METADATA **`, and `# MARKDOWN **` as section markers. They are valid Python files that ruff can parse.

### SQL (sqlfluff)
All `.sql` files under `Fabric/IGDBAnalytics.SQLEndpoint/`. This folder contains schema DDLs (tables, views) for the IGDB Analytics SQL Endpoint on Microsoft Fabric.

---

## What is NOT linted

| Path | Reason |
|------|--------|
| `Labs/` | Tutorial notebooks and early experiments. Not production code. Some notebooks contain SQL magic cells (`-- %%sql`) which are valid Fabric syntax but parse as invalid Python — they would cause `invalid-syntax` errors in ruff. |
| `Docs/exports/` | One-off data exports, screenshots, and query output artifacts from blitz runs. Not code. |
| `Labs/Lab02_Fabric/` | Tutorial notebooks inside Labs — not production code, not linted by ruff either. |

## Personal conventions (intentional, not errors)

### camelCase column names (SQL — CP02 disabled)
All column and table names in this project use camelCase: `gameKey`, `IGDBRating`, `totalReviews`, `sentimentCompound`. This is a deliberate naming convention carried over from the dimensional modeling background of this project. sqlfluff's CP02 rule enforces all-lowercase identifiers — disabled because it would flag every column name as a violation.

### Leading commas (SQL — LT04 disabled, `comma_style = leading`)
SQL is written with leading commas:
```sql
SELECT
     gameKey
    ,gameName
    ,IGDBRating
```
This makes it easier to comment out a column without touching the line above, and produces cleaner git diffs when adding/removing columns. sqlfluff's LT04 rule expects trailing commas — disabled in favour of explicit `comma_style = leading`.

### Uppercase NULL / TRUE / FALSE (SQL — CP04 disabled)
Keywords and literals are written in uppercase per T-SQL convention. sqlfluff CP04 would flag `NULL`, `TRUE`, `FALSE` as violations — disabled.

## Fabric-specific suppressions (not bugs, not style — runtime injections)

### F821 — Undefined names in notebooks
Fabric injects several names into the notebook runtime that do not appear in any import statement:

| Name | What it is |
|------|-----------|
| `spark` | Active SparkSession |
| `display` | Fabric-specific display function |
| `mssparkutils` | Fabric notebook utilities (file system, secrets, etc.) |
| `sc` | SparkContext |

These names are genuinely undefined in the static Python file — they only exist at runtime. F821 is suppressed for all `notebook-content.py` files.

### F811 — Cell re-imports in notebooks
Notebooks can have multiple logical sections, each starting with its own import block. When Fabric exports a notebook to Python, each section's imports appear in sequence in a single file. A name imported in section 1 and re-imported in section 3 triggers F811 (redefinition of unused import). This is structural — it reflects the notebook's cell organisation, not dead code. F811 is suppressed for all `notebook-content.py` files.

### E402 — Imports not at top of file
For the same reason as F811: cell structure means imports can appear anywhere in the file. E402 is suppressed globally.

## sqlfluff and Microsoft-generated DDLs

The SQL DDL files in `IGDBAnalytics.SQLEndpoint/` were exported from SSMS (SQL Server Management Studio) or generated by the Fabric SQL Endpoint tooling. Microsoft's own export format does not comply with sqlfluff defaults:

- **Trailing whitespace** (LT01): SSMS appends trailing spaces to column definitions
- **Indentation** (LT02): SSMS uses tab-style indentation that doesn't match sqlfluff's expected 4-space indent
- **Missing EOF newline** (LT12): SSMS does not append a trailing newline

These are fixed by running `sqlfluff fix` rather than suppressed — they are real fixable issues, just not ones introduced when writing the statements.

LT15 (too many blank lines) is suppressed because SSMS also inserts extra blank lines between blocks, and fixing them is churn with no readability benefit.
