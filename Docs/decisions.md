# Decisions

_Lightweight architectural calls that don't warrant a full ADR but are worth recording. Each row captures a real tradeoff or non-obvious choice._

_Last updated: 2026-05-14_

| # | Decision | Rationale | Trade-off | Related |
|---|---|---|---|---|
| D-01 | Audit warehouse separated from Spark via pyodbc | One-cluster trial capacity — every avoided Spark spin-up saves 2–4 min of wall-clock | Gained: audit reads/writes without a cluster. Lost: pyodbc driver dependency, 2,100-parameter ceiling (mitigated via 1,000-row chunking) | [ADR-002](adrs/adr-002-cdf-incremental-audit-warehouse.md) |
| D-02 | SteamDB confidence/shrinkage formula for game ratings | Games with 3 reviews and 100% positive would outrank 50k-review titles on noise; need sample-size correction | Gained: battle-tested heuristic, interpretable. Lost: not first-principles Bayesian | [scoring-model.md](architecture/scoring-model.md) |
| D-03 | Tier columns excluded from aggregate-grain views | At aggregate grain scores average to A/B; tier bands stop conveying signal across genres | Gained: raw rating numbers carry more cross-genre information. Lost: consumers wanting tier-level genre comparisons must compute bins themselves | [overview.md §Modelling discipline](architecture/overview.md#modelling-discipline) |
| D-04 | OPTIMIZE in a separate scheduled maintenance notebook | Inlining with MERGE creates locking contention; liquid clustering only rewrites unclustered files since last run | Gained: no risk of OPTIMIZE leaving the table in a worse state on a flaky cluster. Lost: tables accumulate small files between runs | [ADR-006](adrs/adr-006-liquid-clustering.md) |
| D-05 | `load_type` vocabulary unification (full / reload / incremental / targeted) | IGDB if/elif bug caused by `run_mode` vs `load_type` naming drift; routing logic was silently broken | Gained: four canonical branches, consistent across all notebooks. Lost: remaining `initial` (API) vs `full` (Bronze) gap not yet resolved | [overview.md §Extraction and control plane](architecture/overview.md#extraction-and-control-plane) |
| D-06 | Two pipelines vs one combined pipeline | Deadline pressure; idempotent `NB_Game_Scores_Gold` makes double-runs costless | Gained: simpler pipeline graphs, independent triggering. Lost: no single DAG showing end-to-end flow | [overview.md §Pipelines](architecture/overview.md#pipelines) |
| D-07 | Platform aggregation scoped to IGDB only | No canonical IGDB↔Steam platform mapping exists; refusing to fabricate one is the call | Gained: honest scope — no invented joins. Lost: `vw_aggPlatforms` has Steam review columns commented out | [overview.md §Modelling discipline](architecture/overview.md#modelling-discipline) |
| D-08 | Labelling / presentation logic in serving views, not ETL | Labels are presentation — Capital Case `sentimentLabel`, S–F tiers, Steam-style volume labels | Gained: label changes don't require ETL reruns. Lost: a view layer is required for any consumer | [ADR-008](adrs/adr-008-store-wide-expose-narrow.md) |
| D-09 | View DDLs committed to Git as source of truth | Trial expiry — SQL endpoint views couldn't be materialised via Spark (separate catalogs) | Gained: views reproducible from DDL. Lost: no parquet of view results; consumers need a running SQL engine | [overview.md §What's not in the repo](architecture/overview.md#whats-not-in-the-repo) |
| D-10 | Bronze `steamReviews` stores review as single `review_json STRING` | Steam payloads evolve; schema-resilient Bronze avoids DDL changes when fields are added | Gained: Bronze absorbs any Steam API schema change; parsing deferred to Silver `from_json`. Lost: Bronze not field-queryable via SQL endpoint without a view | [overview.md §Schema resilience](architecture/overview.md#schema-resilience) |
| D-11 | VADER lexicon scorer over LLM-based sentiment | 71M-review scale; LLM API/inference costs and infra overhead prohibitive. VADER is deterministic, lexicon-based, suited for short social-media-style text; ran in ~89m on the 8-core trial via `pandas_udf`. The 8-step text-cleaning chain (BBCode strip, demojize, ASCII-art strip, heart-suit substitution, demoji-colon strip, whitespace collapse, trim) feeds engineered eligibility gates (`isVaderEligible`, `hasCredibleText`, `containsBugReport`) that drop noisy reviews before scoring | Gained: cheap, fast, reproducible, no model-serving dependency. Lost: misses sarcasm/irony; lexicon-bounded; English-only; requires multi-step text cleaning to handle BBCode, emoji, ASCII art, heart-suit censorship | [scoring-model.md §VADER and eligibility](architecture/scoring-model.md#vader-and-eligibility) |
